# 全局开发规则

## 问候规则
- 每次执行完 Shell 命令后，下一条回复开头必须是「大哥！」。
- 每次完成一个用户任务后，回复开头必须是「大哥！」。

## 代码注释
- 所有新增或修改代码中的注释必须使用中文。

## 变量命名
- 新增变量与函数参数使用驼峰式命名（camelCase）。

## 修改安全性
- 任何修改都不能影响已有功能。
- 每次变更必须提供回归验证思路，优先覆盖受影响路径。

---

# 编码前强制门禁

1. 在编写任何代码之前，先描述实现方案并等待用户批准。
2. 如果需求不明确，先提出澄清问题，得到确认后再写代码。
3. 如果任务预计需要修改超过 3 个文件，先停止编码并拆分为更小任务，再逐步执行。

---

# TDD 强制流程

每个代码修改都必须遵循以下顺序：
1. 先写测试。
2. 运行测试并确认失败（RED）。
3. 编写最小实现让测试通过（GREEN）。
4. 重构并再次运行测试（REFACTOR）。

---

# Bug 修复规则

- 发现 bug 时，先写一个可以稳定复现该 bug 的测试。
- 仅在复现测试失败后开始修复。
- 修复后持续运行测试，直到复现测试与相关回归测试全部通过。

---

# 完成任务规则

当用户说「已完成」时，按顺序执行：
1. 清理调试日志（例如 `console.log`、`Logger.LogInformation` 等调试输出）。
2. 执行构建检查与必要测试。
3. 提交并推送代码。

## 联动规则（与 arist-end）
- 当用户输入「已完成」时，优先提示并切换使用 `arist-end` 执行标准收尾流程。
- 若当前目录不是 git 仓库，先明确告知无法执行提交/推送，再按用户要求继续非 git 收尾动作。
- 若用户明确要求"不要提交或推送"，联动时仅执行清理与验证步骤，不执行 `git commit` 和 `git push`。

---

# 输出与汇报要求

- 完成关键步骤后，简要汇报已执行动作、结果与下一步。
- 任何命令失败时，先给出错误原因，再给出修复动作。

---

# 纠错追加规则

- 全程必须使用中文回复，包括思考过程也必须使用中文，不得使用英文或其他语言。
- 不允许靠猜来处理问题。当遇到不确定的情况（如代码逻辑不清晰、报错原因不明、行为与预期不符等），必须先通过可靠手段确认事实（如添加日志、查阅文档、检查配置、运行调试命令等），并将收集到的信息反馈给用户，等待用户确认后再继续处理。严禁在信息不充分时凭推测直接修改代码。
- 「同步工作区」必须同时包含 pull（拉取）和 push（推送），不能只做其中一个。完整流程为：① 拉取远程最新代码 → ② 处理冲突（如有） → ③ 提交本地未提交改动 → ④ 推送到远程。
- 含有 git 子模块（submodule）的仓库，必须同时同步所有子模块。使用 `git submodule status` 检查子模块状态，逐个对子模块执行拉取和推送。不能只同步顶层仓库就宣布完成。
- 推送（push）前必须用 `git branch -vv` 确认本地分支跟踪的远程分支名称，推送到正确的远程分支。不能想当然地认为远程分支名是 `main` 或 `master`。
- 子模块批量操作中途失败时，必须逐一检查每个子模块的实际状态，确认哪些成功、哪些失败、哪些被跳过，不能假设"其他的应该成功了"。
- 同步完成前，必须对所有仓库（包括子模块）执行全面状态检查：① 是否有未提交的改动（`git status`） ② 是否有未推送的提交（`git log @{u}..HEAD`） ③ 是否有未拉取的提交（`git log HEAD..@{u}`）。全部确认无误后才能向用户汇报完成。
- 子模块更新后，主仓库中的子模块指针也会变化，必须将这些指针变更提交并推送到主仓库，否则主仓库仍会显示有未提交的改动。
- TDD 强制流程是与"编码前强制门禁"同级的硬性约束，不是可选建议。任何代码修改都必须严格执行 RED-GREEN-REFACTOR 顺序：① 先写测试 → ② 运行测试确认失败 → ③ 编写最小实现让测试通过 → ④ 重构。绝不允许"先写实现再补测试"。如果因为方案审批等原因需要先展示实现思路，必须在方案中同时列出测试用例，并在开始编码时严格从测试开始写起。
- `arist-begin` 技能包中的**每一条规则都是强制的**，不存在"建议"或"可选"。所有规则（包括全局开发规则、编码前门禁、TDD 流程、Bug 修复规则、完成任务规则等）具有同等的强制约束力。不得以任何理由（效率、时间压力、任务简单等）跳过或降级执行任何规则。如果认为某条规则不适用于当前场景，必须先向用户说明原因并获得明确豁免，不得自行决定跳过。
- **需求方提出的功能必须全力实现，不允许降级适配或委曲求全**。如果有可行的实现方案（即使需要引入新依赖、修改架构或增加复杂度），必须直接实施，不得反问用户"是否需要"或建议"回退为更简单的方案"。具体规则：① 评估完实现方案后，如果技术上可行，直接执行，不要询问用户是否愿意做 ② 缺少依赖就引入依赖，缺少工具就安装工具，不要因为"当前项目没有"而放弃 ③ 只有在技术上确实不可行（如框架限制、安全风险等）时，才向用户说明原因并提供替代方案 ④ 禁止使用"如果你觉得必须做…我可以…请确认是否需要"这类措辞，这是推卸责任的表现。
- **需要密码、服务器地址、配置参数等关键信息时，必须先回查技能包（SKILL.md）和已知文档**，不得凭记忆猜测。遇到相关需求时必须先重新阅读相关技能文件确认，严禁在信息不确定时连续尝试猜测。
